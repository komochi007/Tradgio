# 用户数据备份与迁移说明

## 1. 当前状态

本文件记录 Tradgio 本地优先方案的数据迁移与恢复规则。任务 39 已实现当前 Origin 内 localStorage 到 IndexedDB 的结构化业务数据迁移；任务 42 才提供跨电脑可操作的备份与恢复界面。在工具完成并通过 Windows 验收前，不应把现有浏览器数据视为已有正式灾难恢复能力。

### 1.1 当前 Origin 内自动迁移

- 用户登录后首次读取业务数据时，系统先为历史无账号记录补齐当前账号，再统一迁移基础资料、三类单据、库存、合同记录和草稿。
- 迁移通过一个 IndexedDB transaction 写入，完成标记为 `local-storage-business-data-v1`，报告包含账号集合和逐 store 记录数。
- 重复执行只读取既有报告，不复制记录；唯一索引冲突、损坏 JSON、数据库版本过新或任一写入失败均不会留下半迁移状态。
- 迁移成功后仍保留全部 localStorage 源数据，任务 40 完成附件 Blob 迁移和后续迁移验收前不得手动删除。
- 当前合同附件仍内嵌 Base64；结构化迁移不等于附件 Blob 迁移，也不提供跨电脑恢复能力。

## 2. 适用场景

- 从 Mac 测试环境迁移脱敏样本到 Windows。
- 更换正式 HTTPS Origin。
- 更换电脑、重装浏览器或重建 Windows 环境。
- schema 高风险升级前保存可恢复快照。
- 日常防范磁盘、浏览器站点数据或电脑损坏。

## 3. 用户必须知道的边界

- 业务数据只存在当前浏览器、当前 Origin 的 IndexedDB 中，不会自动同步到云端。
- 清除站点数据、重装系统、磁盘损坏或更换域名都可能导致当前数据不可访问。
- `.tradgio-backup` 使用用户输入的密码加密；密码不保存，忘记后无法恢复。
- 备份包含全部本地账号、密码校验数据、业务记录和附件，不包含当前登录 session。
- 恢复是整机替换，不支持把两个环境的数据自动合并。

## 4. 备份副本策略

正式使用后至少保留两处副本：

1. Windows 本机之外的移动硬盘或 U 盘。
2. 另一个受控位置，例如加密云盘或另一块离线存储介质。

建议频率：

- 每周至少生成一次完整备份。
- 大批量录入、附件导入、应用升级、Origin 迁移和 Windows 维护前立即备份。
- 每季度至少完成一次恢复演练，确认密码、文件和附件仍可用。
- 仅保留最近一个文件不够；建议保留最近 4 个周期和每月一个长期副本。

备份文件名应包含日期和应用版本，但不得包含客户名、合同号或其他敏感业务内容。

## 5. 标准迁移流程

### 5.1 源电脑

1. 停止录单和附件操作。
2. 执行完整备份并输入不少于 12 个字符的独立密码短语。
3. 确认备份预览中的账号数、记录数、金额、库存和附件统计。
4. 将文件保存到两处位置，并记录文件 SHA-256、应用版本和 schema 版本。
5. 保持源环境不变，直到目标电脑恢复验收完成。

### 5.2 目标电脑

1. 使用支持的 Windows 最新稳定版 Chrome 或 Edge，打开固定正式 HTTPS Origin。
2. 选择备份文件并输入密码。
3. 核对恢复预览，不一致时立即取消。
4. 确认恢复前快照已成功下载。
5. 执行整机恢复，等待应用重开。
6. 重新登录，检查货品、单位、三类单据、合同、库存、搜索和附件下载。
7. 核对恢复报告中的记录数、金额、库存和附件摘要。
8. 验收前不要删除源电脑数据和原始备份。

## 6. 失败处理

- 错误密码或文件损坏：重新选择原始文件，不要反复修改文件内容。
- 版本不兼容：升级到支持该备份格式和 schema 的应用版本后重试。
- 容量不足：清理非 Tradgio 站点数据或扩大可用磁盘空间，重新打开浏览器后重试。
- 恢复中失败：正式数据应保持恢复前状态；保留错误报告和备份文件。
- 恢复后核对失败：停止所有业务写入，使用自动下载的 `.pre-restore.tradgio-backup` 恢复原数据。

## 7. 上线前要求

任务 42 完成真实备份恢复工具后，至少执行两次完整演练。任务 46 必须在全新 Windows 浏览器配置中完成一次恢复，并验证跨 schema 升级。生产就绪清单全部通过前，不得宣称正式数据已有可靠备份能力。
