# AGENT.md

本文件用于约束进入本项目的 AI Coding Agents 行为。

目标：
- 降低架构漂移
- 避免错误假设
- 统一实现风格与决策顺序

## 1. 基本原则

- 全程使用中文输出
- 先确认事实，再提出方案或修改代码
- 需求模糊时先提问，不要猜
- 发现更优路径时主动提出建议
- 当前仓库是已有完整本地 MVP 的可运行代码项目，先读现有实现再修改
- 生产化后续工作当前从任务 35 开始，以生产化路线图为执行依据

## 2. 信息源优先级

发生冲突时，按以下顺序判断：

1. 用户当前明确要求
2. 项目根目录 `AGENTS.md`
3. `ai/AGENT.md`
4. `tasks/production-roadmap.md`（生产化任务）
5. `docs/architecture.md`
6. `docs/PRD.md`
7. 其他文档

如果文档与仓库实际代码冲突：
- 优先以实际代码和用户意图为准
- 但要明确指出文档已过时，并建议同步更新

## 3. 开始工作前必须做的事

- 确认当前任务属于哪个模块
- 确认改动应落在哪个技术分层
- 识别是否涉及库存、导出、合同附件、统一搜索等全局约束
- 检查当前仓库是否已有相关代码或脚手架

如果任务涉及不明确边界，先读：
- `README.md`
- `ai/CONTEXT.md`
- `docs/PRD.md`
- `docs/architecture.md`

如果任务编号为 26-46，还必须读：
- `tasks/production-roadmap.md`
- `tasks/production-readiness-checklist.md`
- `tasks/development-progress.md`

## 4. 编码规则

- 保持原有代码风格
- 默认不加注释，除非逻辑确实不清晰
- 不做无关重构
- 不做风格型改写
- 不随意修改命名体系
- 不引入与当前阶段不匹配的复杂基础设施

修改超过 3 个文件时：
- 先拆成小任务
- 明确每一步目标

## 5. 架构规则

必须遵守：
- 页面层不能直接访问 IndexedDB API
- 页面层不能直接读写附件 Blob store
- 页面层不能直接做导出模板映射
- 所有库存变更必须经过 `Inventory Engine`
- 所有导出必须经过 `Export Service`
- 搜索必须尽量收敛为统一结果结构
- 业务规则应收敛在领域层或 use case 层

不应做的事：
- 直接修改库存快照作为业务真相
- 在页面组件中混入大量业务决策
- 跨模块直接操作别的模块内部状态
- 把外部平台 SDK 调用散落到业务页面

## 6. 文档规则

以下情况应同步更新文档：
- 新增核心目录结构
- 调整模块边界
- 调整关键业务规则
- 改变实现路线或技术方向
- 引入新的环境变量、启动方式、部署方式
- 完成或阻塞生产化任务

不要：
- 把规划写成已完成
- 复制大段 PRD 到 README 或 AI 文件
- 生成与当前仓库状态不符的运行说明
- 只改任务总表而不同步生产化路线图与开发进度

## 7. 任务执行规则

优先顺序：
1. 理解上下文
2. 判断边界
3. 小步实施
4. 自检验证
5. 汇报结果

生产化任务额外规则：
- 按路线图依赖顺序推进，当前下一任务为任务 41
- 任务 34 已由 ADR-0002 锁定 IndexedDB 本地优先方案，不得擅自恢复云端后端路线
- 任务 35-40 已完成；任务 41-42 按依赖顺序实施，完成前不进入 PWA 正式发布
- 任务开始时更新状态为“进行中”，验收完成后才标记“已完成”
- 每项任务同时更新 `tasks/development-progress.md` 和生产就绪清单中的对应证据

修改前：
- 先说明要改什么

修改后：
- 说明改了什么
- 说明为什么这样改
- 说明是否完成验证

如果无法验证：
- 明确说明未验证
- 说明原因

## 8. 禁止事项

- 不要跳过文档直接假设业务规则
- 不要伪造“已运行”“已测试”“已部署”
- 不要无依据新增复杂抽象
- 不要在没有必要时引入新依赖
- 不要为省事而破坏模块边界
- 不要在需求不清时硬做实现

## 9. 面向当前项目的特殊提醒

- 这是单人业务系统，不要提前设计多人协作体系
- 这是桌面优先的 SPA，不要按内容站或 SEO 站设计
- 总览页可以有一定结构设计感，但整体不应营销化
- 列表页、录单页、详情页优先保证效率和可读性
- 当前认证、结构化业务数据和附件 Blob 已切换 IndexedDB；加密备份和离线发布仍未完成
- 未登录旧账号的明文迁移、离线导出和加密备份是当前优先级最高的风险

## 10. 交付标准

一个合格的改动应满足：
- 与文档边界一致
- 没有明显架构越层
- 能被后续 AI 或开发者快速接手
- 说明清楚“已完成什么”和“未完成什么”
- 生产化任务具备自动化验证或明确的未验证说明
- 任务状态、路线图和开发进度保持一致
